Java进击：异常

  • Java进击
  • 常用概念
  • 语法

无论写得多好的程序，总是难免存在着缺陷。对于这些不可避免的错误或者异常，我们总得进行处理，不能够听之任之。本文主要介绍一下Java中的异常处理相关问题，以作备忘。

什么是异常

首先需要说明的是，什么是异常：简单的说，异常就是阻止当前方法或作用域继续执行因素。那么什么时候会发生异常呢？只有在当前环境下的程序无法正常运行下去，也就是说程序已经无法正确运行了，这种情况下，当前环境就会跳出异常。

异常的抛出大概会经历如下几个过程。首先，需要new一个异常对象（我们会在下面介绍一下异常对象的继承关系）；接着该对象会在产生异常的位置终止程序，并且从当前环境中弹出对异常对象的引用；也就是说，从这时起，异常处理机制会接管程序，执行异常处理程序，将程序从错误状态恢复，要么继续执行，要么换种方式执行。

简单来说，异常处理机制就是在程序发生异常时，强制终止程序，并将异常信息反馈给用户。

为什么要使用异常

正如我们导言中所说，异常是无处不在的。为了保证程序的鲁棒性，以及系统的可用性，甚至可以说用户体验，我们都需要加入异常处理机制。

在早期写代码时，我们经常会采用通过函数返回值来判断是否发生了异常，比如说，我们设置一个我们认为不太可能会产生的结果作为一个返回值为我们提供分析，但这样做存在着几个缺陷：

  1. 如异常一样，难免会有意外：我们认为不太可能的返回值可能就是一个正常的程序结果，那么此时就难以分析是否发生异常了；
  2. 可读性会变差：这样做实际上是将正常的逻辑代码与异常处理代码放在一起，降低了代码的可读性。

面向对象编程中，采用异常处理机制是保证代码健壮性的有力方式。采用异常处理机制有如下几个好处：

  1. 降低代码复杂度：在不采用异常处理机制情况下，我们需要在程序中多个位置处理它，这就需要大量的冗余代码，采用异常处理机制后，就可以在一个地方处理异常，节约代码，降低复杂度；
  2. 逻辑分离，各司其职：采用异常处理机制这个过程就是将正常的代码逻辑与异常处理的代码分离，使得代码的阅读、编写以及调试工作井井有条。

异常体系

接下来我们介绍一下异常体系。

Java为我们提供了良好的异常处理机制，我们首先来看一下他们之间的继承关系：

+Throwable
     +Error
          +VirtualMachineError
          +AWTError
     +Exception
          +ClassNotFoundException
          +CloneNotSupportException
          +IOException
               +EOFException
               +FileNotFoundException
               +MalformedURLException
               +UnknownHostException
          +RuntimeException
               +ClassCastException
               +IllegalArgumentException
               +IllegalStateException
               +IndexOutOfBoundsException
               +NoSuchElementException
               +NullPointerException
               +ArithmeticException

从上面的继承关系我们可以看出，Java中异常处理机制均是继承自Throwable类，它有两个子类：一个是Error类，这一类异常是程序无法处理的，比如内存溢出、线程挂掉等等，这一类问题是不可控的，只能由JVM处理，大多数情况下，JVM会选择终止进程；另一类是Exception，这一类属于可处理异常，分为CheckedException（受检异常）和UncheckedException（不受检异常），前者发生在编译阶段，后者发生在运行阶段。

关于异常处理的几个认识

Java的异常类很多，产生原因以及处理方式也千变万化，这里就不一一而足了。以下是根据不同的异常总结出来的几点认识，和大家共享，有不到之处，请指教。

  1. 尽量完善finally代码块。finally代码块会在try代码块return前调用，这个过程我们就可以通过finally做些异常处理的后续工作，比如说，在读写文件异常时，后续释放资源的操作是没有执行的，这样的关闭资源操作就可以放在finally中执行，这样可以正确、有效的释放占用的资源。
  2. 对于程序中没有显示声明的异常，当异常抛出时，通常显示异常出处，而对于已经显示声明了的某个异常，抛出时则不会显示出处。
  3. 尽可能减少try代码块。我们在写代码过程中，为了简单，为了代码的美观，通常会将大段的代码放在try代码块中，这样的坏处就是当异常发生时我们不能够很准确的定位到异常位置，也无法为不同的可能发生异常的地方做针对性处理。尽可能减少try代码块可以很好的解决上述问题。
  4. 尽可能完善catch语句的异常类型。和上面的问题一样，我们通常为了简单，会直接将catch块指定为Exception类，这个指定涵盖的范围太光了，我们应尽量对不同的预期异常位置进行针对性处理。
  5. 对于捕获的异常进行适当的处理，不要不予理睬。
  6. 异常链的采用，通常可以提高代码的可理解性、系统的可维护性和友好性。关于这一点。
