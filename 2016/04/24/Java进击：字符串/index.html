Java进击：字符串

字符串是任何一个语言都无法规避的一个问题。在Java中，分别有三个类用于处理字符串，而他们之间的区别也使得很多人云里雾里。本文将主要介绍一下String、StringBuffer、StringBuilder及其异同，以作备忘。

String

我们首先需要明确的是，对于Java来说，String并不是一个基本数据类型，实际上它是一个对象。这也是为什么我们经常将String和数组之类的放在一起说的原因。String类是final型的，也就是说它既不能改变也不能被继承。说到这里，可能就已经有人开始迷惑了：不对啊，我每次都是直接给String赋值使其改变的。只能说被JVM骗了，实际上，每次对String对象的操作，都创建了一个新的String对象。

接下来我们需要讲一下String对象初始化的问题。String对象初始值为null，但在这个阶段并没有new String对象，也就是说没有创建对象的引用，也没有分配内存空间。与null相比，容易混淆的是“”与new String()。“”与new String()是等同的，他们都表示new了一个String对象，但其值为空，也就是说创建了对象的引用，分配了内存空间，只是没有赋值，或者说值就是空。

Java字符串有个特殊的概念就是字符串池，这个概念我们在“JVM内存区域介绍”中提到过，简单说，就是为了提高效率，减少内存空间的占用，创建每个字符串对象时，先检查字符串池中是否存在相等的字符串，如果存在，就不再创建，直接返回字符串池中对该对象的引用，如果不存在则创建，并将其放入字符串池中，返回新建对象的引用。这也是为什么推荐字符串采用直接赋值的方式，而不是每次都new。

StringBuffer

StringBuffer与String都可以用来存储字符串，但由于实现方式的不同，就出现了适用范围的不同。二者对比如下：

  • StringBuffer不能采用赋值的方式进行初始化，它必须要实例化，也就是说必须通过构造方法进行初始化，而String则没有这方面的约束；
  • StringBuffer在进行字符串修改时，并不会产生新的字符串对象，而String除了上面所说的常量赋值外，均会产生新的String对象；就这一点来说，StringBuffer在内存使用方面是优于String的；
  • 再就是适用范围的不同，StringBuffer更加侧重于字符串的变化，比如追加、修改、删除等。

根据上面所说，我们建议在频繁的字符串变动时，尽量采用StringBuffer，性能会提升很大一截。

StringBuilder

StringBuilder也是可变的字符串对象，相对于StringBuffer来说，最大的不同在于StringBuilder是线程不安全的，基于这点，其速度要比StringBuffer快。
